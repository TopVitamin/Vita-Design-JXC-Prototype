# 强盛进销存前端原型

一个基于React+TypeScript+Vite实现的进销存中后台原型项目，用于演示**批发、零售、库存、往来**等一期核心业务在前端界面的承接方式。

这个项目的重点不是“做一套可直接上线的ERP系统”，而是把产品方案里的核心页面、菜单结构和业务主链路，先落成一个**可浏览、可演示、可继续扩展**的原型Demo。

## 项目定位

这个原型主要服务3类场景：

- 产品方案演示：把进销存一期范围讲清楚
- 教学示例：让学员看到业务方案如何下沉为前端原型
- 后续设计/开发前置：为UI细化、交互补全、前后端实现提供页面骨架

当前原型围绕“强盛科技2014-2015渠道扩张期”的业务背景展开，核心目标是演示以下主链路：

- 渠道批发接单与订单流转
- 门店零售收银
- 库存共享底账查询
- 财务收款登记与客户往来查询

## 当前包含什么

### 1. 核心演示页面

当前已经重点承接的核心页面包括：

| 页面 | 说明 |
| --- | --- |
| 工作台 | 展示主入口、重点模块和演示主线 |
| 销售订单 | 支撑批发开单、列表查询、详情、编辑、新增 |
| 零售收银 | 展示门店现场成交的收银操作界面 |
| 库存查询 | 展示现存、占用、可用库存口径 |
| 收款登记 | 展示客户回款录入流程 |
| 客户往来查询 | 展示客户应收、回款记录和余额视图 |

### 2. 次级页面

项目同时补了较完整的一级/二级菜单结构，包含但不限于：

- 基础资料：商品管理、客户管理、供应商管理、仓库管理
- 销售管理：销售出库、销售查询
- 采购管理：采购订单、采购入库、采购退货
- 库存管理：调拨管理、盘点管理、报损管理
- 往来管理：应收查询、应付查询、付款登记
- 查询统计：销售汇总、库存余额
- 系统设置：用户与权限、单据编号、期初初始化、打印模板、操作日志

这些页面当前按3种深度组织：

- `core`：重点演示页，内容更完整
- `secondary`：结构清晰、信息完整，但不追求深交互
- `placeholder`：保留入口和定位，用于体现一期全貌

## 项目特点

- 不是只做单页，而是先把完整菜单和后台骨架搭起来
- 重点页面有明确主线，不是静态“页面拼盘”
- 使用统一导航、页面元信息和模块契约，便于后续扩展
- 部分页面通过通用列表页、表单页、查询页、配置页方式生成，减少重复搭建成本
- 全部数据当前使用前端Mock数据，方便教学、演示和快速迭代

## 目录结构

```text
jxc-prototype
├── src
│   ├── app           # 路由、导航、页面元信息
│   ├── components    # 应用外壳、通用UI组件、辅助组件
│   ├── contracts     # 模块契约层（一个模块一个文件）
│   ├── data          # 收口注册层、局部页面专属数据
│   ├── mocks         # 核心页面直接消费的 Mock 数据入口
│   ├── pages         # 页面级原型
│   ├── docs          # UI 设计与实现说明
│   ├── utils         # 工具函数
│   ├── App.tsx       # 应用壳入口
│   └── main.tsx      # 应用启动入口
├── index.html
├── package.json
├── AGENTS.md
└── README.md
```

### 当前分层说明

- `app/`
  负责 `ViewKey`、导航结构、页面元信息、路由注册
- `contracts/modules/`
  负责业务模块契约，是当前模块定义的主编辑区
- `data/modulePages.ts`
  负责聚合并导出模块定义，不再长期承载完整模块内容
- `mocks/`
  给工作台、销售订单、库存查询、收款登记等核心页面提供更清晰的 Mock 入口

## 当前维护方式

当前工程已从“单一大文件维护全部模块定义”的方式，调整为“一个模块一个契约文件”的方式。

推荐维护顺序：

1. 先确认模块 PRD、页面 Demo PRD、字段清单
2. 再修改 `src/contracts/modules/` 下对应模块文件
3. 如涉及导航或页面类型，再补 `src/app/navigation.ts`
4. 如涉及路由，再补 `src/app/routes.tsx`
5. 最后执行 `npm run build`

### 页面分层

当前页面按 3 种深度组织：

- `core`：重点演示页，单独实现，承载主链路
- `secondary`：结构完整，优先由通用骨架承接
- `placeholder`：保留入口和定位，后续再深化

### 典型页面归类

- `core`
  工作台、销售订单、零售收银、库存查询、收款登记、客户往来查询
- `secondary`
  商品管理、客户管理、供应商管理、仓库管理、采购订单、采购入库、采购退货、销售出库、调拨、盘点、报损

## 适合怎么使用

如果你是学员，可以把它当成一个“产品文档如何落到前端原型”的参考样例，重点看：

- 菜单结构怎么从业务范围里抽出来
- 核心页和次要页怎么分层
- 如何用一套统一骨架承接多个业务模块

如果你是外部读者或面试官，可以把它理解成：

- 一个有明确业务背景的进销存一期原型
- 一个强调信息架构和业务主链路的中后台Demo
- 一个适合继续往UI细化、交互细化和真实开发推进的前端起点

## 当前边界

这个项目当前是**原型工程**，不是完整业务系统，因此有几个边界需要明确：

- 目前没有接入真实后端接口
- 当前数据为Mock数据，不代表真实数据库设计
- 并非每个页面都做了完整交互闭环
- 部分页面仍以结构展示和信息占位为主
- 权限、审批、打印、初始化等能力目前主要用于展示入口与扩展方向
- `contracts/modules/` 中的契约优先服务前端演示，不等于最终后端数据模型

如果你希望继续往下做，推荐的顺序是：

1. 先补核心页面交互细节
2. 再补页面之间的单据联动关系
3. 再接入真实接口和状态流转
4. 最后再做权限、打印、初始化、日志等系统级能力

## 后续可扩展方向

- 接入真实后端接口和持久化数据
- 完善销售、采购、库存、往来之间的单据联动
- 补齐状态机、异常态、空态和权限态
- 从中保真原型继续升级为高保真业务系统前端

这个仓库更适合作为**教学样例、产品演示样例、原型基线工程**来理解，而不是把它当成“现成可商用的进销存系统”。
