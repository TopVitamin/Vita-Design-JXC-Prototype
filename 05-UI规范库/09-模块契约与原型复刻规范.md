# 模块契约与原型复刻规范

## 一、文档目的

本文件用于沉淀“从字段清单 / PRD / Demo PRD 到前端原型”的复刻规则。  
目标不是讲设计理论，而是让后续新增模块时可以直接复用当前已经跑通的做法，减少反复试错。

适用场景：

- 新增一个主数据模块
- 新增一个单据模块
- 基于已有字段清单和 Demo PRD 快速产出前端 Demo
- 对齐产品文档和前端实现口径

---

## 二、信息源优先级

前端实现必须按以下顺序服从信息源：

1. 字段清单
2. 主PRD
3. 前端Demo版PRD
4. 当前项目 `AGENTS.md`
5. 当前工程代码结构
6. UI规范库

如果冲突，优先级为：

`字段清单 > 主PRD > 前端Demo版PRD > 代码现状 > AI默认习惯`

补充说明：

- 字段清单定义字段本体、枚举、必填性、展示位置
- 主PRD定义业务边界、动作规则、异常和引用影响
- Demo PRD定义页面结构、控件、显隐、反馈和页面分组

## 三、模块落地原则

### 3.1 一个模块一套文档组

后续模块文档建议统一按下面结构维护：

```text
模块目录/
├── [模块]字段清单.md
├── [模块]主PRD.md
└── [模块]_前端Demo版PRD.md
```

不要再把多个主数据或多个单据强行合并到一个 PRD 里，否则后续新增模块时会反复改总文档。

### 3.2 一个模块一个契约文件

前端契约层统一按下面方式维护：

```text
src/contracts/modules/
├── productManagement.ts
├── warehouseManagement.ts
├── purchaseOrders.ts
└── ...
```

规则：

- 一个模块一个文件
- `modulePages.ts` 只负责聚合注册，不再长期承载完整模块定义
- 复用逻辑统一放在 `shared.ts`

### 3.3 页面分层

页面按复杂度分为：

- `core`：核心业务页，单独实现
- `secondary`：标准管理页，优先由 Generic 骨架承接
- `placeholder`：只保留入口和定位

新增模块前先判断页面属于哪类，不能所有页面都按独立大页硬写。

## 四、主数据模块复刻规范

适用模块：

- 商品
- 客户
- 供应商
- 仓库

### 4.1 通用页面结构

每个主数据模块统一包含：

- 列表页
- 查看页
- 新增页
- 编辑页

### 4.2 通用动作

统一支持：

- 查看
- 编辑
- 启用
- 停用

统一不支持：

- 删除

标准口径：

`主数据不提供删除能力。若对象不再使用，统一通过“停用”处理；已被业务单据引用的数据必须长期保留，确保历史记录可追溯。`

### 4.3 通用字段规则

- 编码类字段新增可编辑
- 编码类字段一旦被下游引用后只读
- 停用原因仅在停用时展示
- 系统信息在查看页和编辑页可读，不在新增页展示
- 空值统一展示为 `-`

### 4.4 详情页分组建议

默认顺序：

1. 基础资料
2. 经营 / 控制信息
3. 系统信息

备注字段是否单独成卡片，必须以 Demo PRD 为准，前端不得自行强化布局。

## 五、前端交互复刻规范

### 5.1 成功反馈

成功提示统一使用全局顶部居中 `Message.success`：

- 自动消失时间：`2s`
- 不使用页面内插入式成功提示条

### 5.2 错误 / 警告反馈

- `Message.error` / `Message.warning`
- 自动消失时间：`3s`

### 5.3 TextArea 规则

- `TextArea` 默认高度按单行展示
- 输入增多时自动增高
- 不默认渲染成大块文本域

适用场景：

- 备注
- 地址
- 停用原因

### 5.4 启停用交互

- 启用 / 停用操作使用列表页操作列按钮
- 停用后需要同步刷新列表状态
- 成功后显示顶部 toast

## 六、数据与缓存复刻规范

### 6.1 新增页初始化

新增页默认使用**空白草稿**，只保留必要默认值：

- 主数据默认：`状态 = 启用`
- 单据默认：`状态 = 草稿`

禁止：

- 克隆第一条 Mock 数据作为新增默认值
- 预填现成业务数据冒充新增

### 6.2 本地缓存

当前原型在无后端前提下，统一使用 `localStorage` 做正式保存缓存。

要求：

- 列表页优先读取缓存数据
- 详情页优先读取缓存数据
- 编辑页优先读取缓存数据
- 内置 Mock 作为初始种子数据

适用范围：

- 主数据新增/编辑
- 单据新增/编辑
- 启用/停用状态更新

### 6.3 草稿缓存

草稿缓存属于增强能力，不是当前所有模块的强制要求。

优先级建议：

1. 先做正式保存缓存
2. 再做草稿自动缓存
3. 最后补离开页面未保存提醒

## 七、快速复刻步骤

后续新增模块时，建议按下面顺序做：

1. 确认字段清单、主PRD、Demo PRD 都已齐全
2. 在 `src/contracts/modules/` 新建对应模块契约文件
3. 在 `src/data/modulePages.ts` 注册聚合
4. 判断是否走 Generic 页面还是独立页面
5. 核对成功反馈、错误反馈、TextArea、启停用、缓存逻辑是否符合本规范
6. 执行 `npm run build`

---

## 八、验收清单

- [ ] 模块是否有独立字段清单、主PRD、Demo PRD
- [ ] 是否有独立契约文件
- [ ] 是否没有把完整定义堆回 `modulePages.ts`
- [ ] 新增页是否为空白草稿而非克隆 Mock 数据
- [ ] 保存后是否写入本地缓存
- [ ] 成功提示是否为顶部居中 2s toast
- [ ] 错误/警告是否为 3s toast
- [ ] TextArea 是否默认紧凑而非大块展开
- [ ] 已通过 `npm run build`
